## このスキルが解決する問題

技術的な知見は、解決した瞬間にしか正確に書けない。後で「あの時どうやって直したか」を思い出すコストは、その場で3行書き残すコストより圧倒的に高い。一方で、テンプレートが面倒だと誰も書かなくなる。このスキルは「会話の文脈から半自動で下書きを作り、ユーザーには確認と差分修正だけさせる」ことで、キャプチャの摩擦をゼロに近づけることを品質要件と同格に扱う。

## ワークフロー

### 1. 会話の文脈から下書きを自動生成する(質問より先に下書き)

ユーザーに「何を書けばいいですか?」と白紙から聞かない。会話履歴を読み返し、以下を素材としてそのまま使う:

- エラーメッセージは**原文ママ**(意訳・要約しない。検索に引っかからなくなるため)
- 実際に叩いたコマンドとその出力
- 試して効かなかったアプローチ
- 最終的に効いた解決策/決定/結論

これらから type ごとのテンプレート(後述)を埋めた下書きを作り、「この内容で残しますが、認識合っていますか?」とユーザーに見せる。ゼロから書かせるのではなく、差分を直させる形にすることが重要。

### 2. type を確定する

3種類のうちどれかに分類する。会話から自明な場合は確認なしで進めてよいが、判断が割れる場合は **AskUserQuestion** で一度だけ聞く。

| type | 判断のヒント |
|---|---|
| `troubleshoot` | 起点がエラー・不具合・「動かない」だった |
| `decision` | 複数の選択肢を比較して1つを選んだ(ライブラリ選定、設計方針、トレードオフの判断など) |
| `research` | 「調べてみた結果こうだった」という調査・検証が主目的で、エラーも決定もない |

会話の中に**複数の独立した学びが混在している場合は、1つのファイルに詰め込まず、type ごと・トピックごとに分割する**(アトミック性。理由は後述)。

### 3. scope を確定する

判定基準は1つだけ:「**このプロジェクト以外でも役立つ知見か?**」Yes なら `user`、その場限り(このリポジトリ固有の設定・このサービス固有の障害など)なら `project`。

迷ったら **AskUserQuestion** で確認する。よくある誤判定:
- 「Nixのoverride/overrideAttrsの挙動」のような言語・ツールの一般知識 → `user`
- 「このdotfilesリポジトリのos/gui.nixでこう直した」のような、このリポジトリの設定値そのもの → `project`
- 同じ会話の中にこの両方が混ざっていることはよくある。その場合は遠慮せずファイルを分割する。

`scope: project` の場合、プロジェクトルートは現在の作業ディレクトリから上方向に `.git` または既存の `.claude` ディレクトリを探して特定する(後述のスクリプトが自動でやる)。見つからない場合はユーザーに確認する。

### 4. 重複・関連を確認する

新規ファイルを作る前に、既存のナレッジを軽く検索する:

```bash
grep -ril "<キーワード>" ~/.claude/knowledge 2>/dev/null
grep -ril "<キーワード>" <project-root>/.claude/knowledge 2>/dev/null
```

- ほぼ同じトピックの既存ファイルが見つかったら、新規作成ではなく**そのファイルの更新**(`updated`/`last_verified` の打ち直しと内容の追記)を提案する。ナレッジが分裂すると検索性が落ちる。
- 内容は別だが関連する既存ファイルが見つかったら、新規ファイルの `related` frontmatter と本文の参照セクションに **@メンション**でリンクする(次節)。

### 5. @メンションでの相互リンク

ナレッジファイル同士をリンクするときは、`@` + **絶対パス**を使う。

```
related: ["@/home/chomo/.claude/knowledge/research/2026-06-10-nix-override-composition.md"]
```

```markdown
## 参照 (References)
詳しい検証は @/home/chomo/.claude/knowledge/research/2026-06-10-nix-override-composition.md を参照。
```

相対パスを使わない理由: ナレッジファイルは「どのディレクトリから読まれるか」が保証されていない(別プロジェクトから読まれることもある)。自己完結性の原則と同じ理由で、参照先は常に一意に解決できる絶対パスにする。`related` には知見ファイルへの@メンションだけでなく、PR・Issue・外部ドキュメントのURLも同様に含めてよい。

### 6. ファイルを生成する

このSKILL.mdが置かれているディレクトリ(スキル起動時に示される "Base directory for this skill")を `<skill-dir>` として、付属スクリプトでファイルを生成する:

```bash
bash <skill-dir>/scripts/scaffold.sh \
  --type troubleshoot \
  --scope project \
  --slug gdm-hyprland-uwsm-session-duplicate \
  --title "GDMにHyprland(uwsm-managed)が重複して表示される" \
  --tags "nixos,hyprland,gdm" \
  --confidence high
```

スクリプトが frontmatter とtype別の見出し骨格を埋めたファイルを作成し、絶対パスを標準出力に返す。その後 **Edit ツールでプレースホルダのコメント(`<!-- ... -->`)を実際の内容に置き換える**。コメント自体は埋め終わったら削除する。

主なオプション(詳細は `bash <skill-dir>/scripts/scaffold.sh --help`):

| オプション | 説明 |
|---|---|
| `--type` | `troubleshoot` / `decision` / `research` (必須) |
| `--scope` | `user` / `project` (必須) |
| `--slug` | ファイル名・idに使うkebab-caseのスラッグ(必須。ASCII小文字とハイフンのみ) |
| `--title` | 人間向けタイトル(必須。日本語可) |
| `--subdir` | `knowledge/` 配下のサブディレクトリ(例: `nixos`, `decisions`, `aws`)。未作成でも自動作成される |
| `--tags` | カンマ区切り。**検索する人が打ちそうな語**で書く(社内用語・内部コードネームではなく) |
| `--root` | scope=project のときのプロジェクトルートを明示指定(省略時は自動探索) |
| `--related` | カンマ区切りの@メンション/URL |
| `--confidence` | `high`(実証済み) / `medium`(おそらく正しいが未検証) / `low`(仮説) |
| `--status` | 既定 `draft`。後で検証して問題なければ `validated` に上げる運用を想定 |

`id` は ULID ではなく `--slug` の値をそのまま使う。ULIDは生成に乱数が要るうえ人間が読んでも何の話か分からず、grepでの検索性を優先したほうが実用上勝るため。

### 7. サブディレクトリの選び方

`knowledge/` 配下が肥大化したら細分化してよい。迷ったら:

1. まず既存の `knowledge/` を `ls` して、既にあるサブディレクトリの分け方に倣う(プロジェクトごとに流儀が育つので、新規に独自分類を増やさない)
2. 何もなければ: `decision` は `decisions/` にまとめる(ADRは時系列で一覧したいことが多い)。`troubleshoot`/`research` はドメイン名(`nixos/`, `aws/`, `docker/` など)で分ける(困って検索するときはドメインで探すことが多い)

### 8. 保存前の最終確認

生成したファイルの内容(特に「解決策」や「決定」の本文)をユーザーに見せ、間違いがないか確認してもらう。レビューを長々と要求するのではなく、「これで保存します。直したい点があれば教えてください」程度の軽い確認でよい(摩擦ゼロの原則)。

## 共通frontmatterスキーマ

```yaml
title: ""             # 検索する人が実際に打ちそうな言葉で書く
id: ""                 # --slug と同じ値
type: troubleshoot|decision|research
scope: user|project
created: YYYY-MM-DD     # 実行時の日付。`date +%Y-%m-%d` で取得し、勝手に書かない
updated: YYYY-MM-DD      # 新規作成時は created と同じ
last_verified: YYYY-MM-DD # 新規作成時は created と同じ。後で「まだ正しいか」を確認したらこれだけ更新する
status: draft|validated|deprecated|archived
author: ""
tags: []
environment: ""        # 対象バージョンなど(例: "NixOS 26.11 unstable, Hyprland 0.55.4")
confidence: high|medium|low
related: []             # 他ナレッジへの@メンション、PR/Issue/URL
```

`last_verified` を `created` より重視する。`created` だけでは「この情報がまだ正しいか」は分からない。同じ内容を再確認した・再現した際は、本文を変えなくても `last_verified` だけ today に更新する運用にする。

## typeごとの本文テンプレート

### troubleshoot

```markdown
## 問題 (Problem / Symptom)
## 原因 (Root Cause)
## 解決策 (Resolution)
## 試したが効かなかったこと (Dead ends)
## 根拠・参照 (Evidence)
```

- **問題**: エラーは原文ママ。再現条件(バージョン、コマンド、環境)も書く。
- **原因**: 5 Whysで深掘りする。表面的な原因で止めない。**「確証」か「仮説」かを明記する**(検証していないのに確定的に書くと、後で読んだ人が誤った前提で動いてしまう)。
- **解決策**: コピペで再現できる具体的なコマンド・設定をそのまま書く。「〜を直した」のような要約だけでは再現できない。
- **Dead ends**: 試したが効かなかったことを残す。これがないと、次に同じ問題に当たった人(未来の自分含む)が同じ罠を踏む。

### decision (ADR風)

```markdown
## 背景・課題 (Context)
## 決定 (Decision)
## 検討して捨てた選択肢 (Alternatives considered)
## 影響・トレードオフ (Consequences)
## 根拠・参照 (Evidence)
```

- **検討して捨てた選択肢**: 採用した案だけでなく、**なぜ他の案を採らなかったか(why not)**を書く。これがないと、後で「あの案の方が良かったのでは」と同じ議論を繰り返すことになる。

### research

```markdown
## 問い (Question)
## 結論 (Findings)
## 詳細・根拠 (Details)
## 未解決・次の問い (Open questions)
## 参照 (References)
```

- **結論を先に書く**。詳細を読まなくても結論だけで判断できるようにする(Diátaxisのreference的な発想)。

## 執筆原則

- **アトミック性**: 1ファイル1問題/1決定/1問い。複数の学びを1ファイルに混ぜると、検索でヒットしても関係ない情報も一緒に読まされ、再利用性が落ちる。
- **自己完結性**: 「上記の」「前述の」のような、ファイル単体では意味が取れない外部依存を書かない。RAGでは断片(チャンク)単位で検索にヒットするため、そのファイルだけ読んで意味が通る必要がある。
- **検索者の言葉**: title と tags は、困っている人が実際に検索バーやチャットに打ちそうな語で書く。社内用語・内部のコードネームより、エラーメッセージの一部や一般的な技術用語を優先する。
- **delta原則**: 公式ドキュメントに書いてあることを再掲しない。公式docに**載っていない**落とし穴・自社/個人固有の設定・回避策だけを書く。再掲は陳腐化したときに二重にメンテコストがかかる。
- **confidenceの区別**: 実証済みのことと仮説を同じトーンで書かない。`confidence: low` で書いておけば、後で誤りだと分かっても「あれは仮説だった」と分かる。

## 振る舞いの補足

- `created`/`updated`/`last_verified` は必ず `date +%Y-%m-%d` で実行時の日付を取得して使う(会話の中の日付表現や記憶に頼らない)。
- ファイル名は既定で `{created}-{slug}.md`(スクリプトが自動でこの形式にする)。
- 保存先ディレクトリが未作成でも自動作成してよい(スクリプトが `mkdir -p` する)。
- 1回の会話で複数のナレッジを作る場合、scaffoldスクリプトをファイルごとに繰り返し呼び出す。
